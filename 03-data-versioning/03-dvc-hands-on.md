## 🏗️ Infrastructure — S3 Bucket via Terraform

Instead of manually clicking through the AWS console, the S3 bucket for DVC remote storage is provisioned using **Terraform** (IaC — Infrastructure as Code).

The `terraform/` directory uses the official `terraform-aws-modules/s3-bucket/aws` module and includes:
- Versioning, SSE encryption, public access block
- Lifecycle rules (IA → Glacier transitions, version expiry)
- Access logging, HTTPS-only bucket policy
- Remote state stored in S3 + DynamoDB locking

```bash
cd terraform/
terraform init
terraform apply     # provisions the S3 bucket
```

Once the bucket is created, use its name in the DVC remote config:
```bash
dvc remote add -d wineremote s3://<bucket-name>
```

python3 -m venv .venv

```bash
python3 -m pip install dvc dvc_s3
```
- `dvc` — core DVC package
- `dvc_s3` — S3 backend support (required for `dvc push/pull` with AWS)

### 4. Initialize Git & DVC
```bash
git init
dvc init
```
> `dvc init` works just like `git init` — creates a `.dvc/` directory to manage DVC config and cache.

## 📦 Tracking a Dataset with DVC

### Add dataset to DVC tracking
```bash
dvc add data/wine_sample.csv
```

This does two things:
1. Creates `data/wine_sample.csv.dvc` — a small metadata file containing:
   - The **filename**
   - A **unique MD5 checksum** of the current version
2. Adds `data/wine_sample.csv` to `.gitignore` — so Git never sees the actual data

### Inspect the .dvc file
```bash
cat data/wine_sample.csv.dvc
```
```yaml
outs:
  - md5: a1b2c3d4...        # checksum of this version
    size: 1234
    path: wine_sample.csv
```

## ☁️ Connecting DVC to S3

### Create an S3 bucket
- Go to AWS Console → S3 → Create bucket
- Give it a unique name (e.g. `mlops-demo-abhishek-dvc-bucket`)

### Configure AWS credentials on your CLI
```bash
aws configure
# Enter: Access Key ID, Secret Access Key, Region, Output format
```
> ⚠️ Never store AWS credentials in any file or push them to Git.

### Add S3 as DVC remote
```bash
dvc remote add -d wineremote s3://bucketname/foldername
```
- `-d` sets this as the **default** remote
- This saves the remote URL to `.dvc/config`

### Push data to S3
```bash
dvc push
```
The file is stored in S3 at:
```
bucketname/foldername/files/md5/<first-2-chars-of-checksum>/<remaining-checksum>
```
Each unique checksum = a unique version of the dataset stored in S3.

## 🔄 The Full Git + DVC Workflow

```
┌─────────────────────────────────────────────────────┐
│                  Your Local Machine                  │
│                                                      │
│   wine_sample.csv  ──dvc add──►  wine_sample.csv.dvc │
│         │                               │            │
│         │                               │            │
│    dvc push                         git push         │
│         │                               │            │
│         ▼                               ▼            │
│      S3 Bucket                    Git Repository     │
│  (actual data)                  (.dvc file, config)  │
└─────────────────────────────────────────────────────┘
```

### Commit metadata to Git after every change
```bash
git add data/wine_sample.csv.dvc .dvc/config
git commit -m "Track dataset v2 with DVC"
git push
```

## 📝 Versioning Workflow — Step by Step

| Step | Action | Command |
|---|---|---|
| 1 | Make changes to CSV | (edit in vim/editor) |
| 2 | Re-track with DVC | `dvc add data/wine_sample.csv` |
| 3 | Commit updated `.dvc` file | `git add data/wine_sample.csv.dvc` |
| 4 | Push metadata to Git | `git commit -m "..."` + `git push` |
| 5 | Push new data version to S3 | `dvc push` |

> Every time the CSV changes, the MD5 checksum in the `.dvc` file changes → a new version is stored in S3.

## 👥 How Teammates Pull the Right Version

When a colleague clones the repo:
```bash
git clone <repo-url>
cd wine-prediction-demo
dvc pull        # downloads the exact version referenced in .dvc file
```

- `.dvc/config` tells DVC **where** the data is (which S3 bucket)
- `wine_sample.csv.dvc` tells DVC **which version** to pull (via checksum)
- **Git is the source of truth** — whatever checksum is in Git = the version everyone should use

## 💡 Key Takeaways

| Stored in Git | Stored in S3 |
|---|---|
| `wine_sample.csv.dvc` | `wine_sample.csv` (actual data) |
| `.dvc/config` (remote URL) | All historical versions of the CSV |
| `train.py`, source code | — |

- S3 may hold **100 versions** of a dataset — the `.dvc` file in Git is always the **source of truth**
- Never push the actual dataset to Git
- Always commit `.dvc` files and `.dvc/config` to Git after every data change
